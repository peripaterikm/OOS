## Contract Content Summary

### Sections Defined

1. **Purpose** — Why output modes exist (CP1251/CP1252 vs UTF-8 terminals), why ASCII-safe is default, why UTF-8 is opt-in.
2. **Output Modes** — `ascii_safe` (default), `utf8` (opt-in), unknown mode fail-closed, no automatic encoding detection.
3. **CLI Policy** — Commands requiring `--utf8` (3), commands to audit (1), commands excluded (14), expansion policy.
4. **Symbol Mapping Table** — Canonical ASCII-safe and UTF-8 symbol mappings with code points and usage context.
5. **Compatibility Rules** — ASCII-only guarantee, JSON artifact independence, Markdown artifact policy, business logic independence, CP1251/CP1252 compatibility.
6. **Testing Requirements** — ASCII-safety checks, Unicode presence checks, CP1251/CP1252 compatibility, no regression, minimum 12 tests.
7. **Non-Goals** — 9 explicit exclusions.
8. **Acceptance Criteria for Item 1.2** — 12 criteria item 1.2 must satisfy.
9. **Motivation** — v2.8 item 4.1 gap documentation.
10. **References** — Links to all related files.

### Key Decisions Recorded

| Decision | Rationale |
|----------|-----------|
| ASCII-safe is default | Safe for all Windows terminals without user configuration |
| `--utf8` is single boolean, all-or-nothing | Unambiguous, idempotent |
| No encoding auto-detection | Platform-specific, non-deterministic |
| Unknown mode fails closed | Prevents silent fallback to wrong mode |
| `[CORRECTED]` unchanged in both modes | Semantic label, not decorative |
| JSON artifacts mode-independent | Structure unaffected by output mode |
| Only 3 commands get `--utf8` | Minimal surface area; expansion requires justification |
