# 6.2 Isolated AI Council Critique

## Goal

Add isolated, provider-style council critique for top scored ideas, including suspiciously clean protection and founder manual-review safeguards.

## Context

Roadmap 5.2 introduced weighted ideation scores and council-candidate thresholds. Roadmap 6.1 added cheap deterministic anti-pattern checks. This item adds the next layer: semantic critique contracts for selected top ideas only, without live LLM/API calls or orchestrator wiring.

## Inputs

- Scored ideas from `src/oos/ideation_mode_comparison.py`
- Idea-like records from pattern-guided or compatible ideation outputs
- Stubbed, isolated role-provider payloads
- Common AI metadata contract from `src/oos/ai_contracts.py`

## Outputs

- `src/oos/ai_council_critique.py`
- `tests/test_ai_council_critique.py`
- `tests/test_ai_council_critique_acceptance.py`
- README documentation for isolated council critique
- Roadmap and Dev Ledger state updates after green validation

## Key decisions

- Council critique remains standalone and is not wired into `run-signal-batch`.
- Each council role has its own provider boundary before aggregation.
- Standard mode caps critique at 3 ideas per opportunity.
- Missing or invalid role output marks `critique_unavailable` and requires founder manual review.
- Founder decision authority remains explicit and final.

## Alternatives rejected

- Combined all-role critique prompt: rejected because the roadmap requires isolated role outputs.
- Running critique on every idea: rejected because council should only run on top ideas.
- Treating a clean critique as a pass: rejected because `suspiciously_clean` requires founder challenge.
- Live provider calls: rejected until provider execution, cost controls, and operational gates are explicitly in scope.

## Validation

- Focused council tests verify selection, max-per-opportunity behavior, isolated role calls, validation, fallback, suspiciously clean protection, unsupported claims, founder authority, and no live API calls.
- Acceptance tests verify no orchestrator wiring, isolated-role architecture, suspiciously clean protection, founder final authority, and active roadmap advancement.
- Full unittest discovery, `scripts/oos-validate.ps1`, and `scripts/verify.ps1` are required before commit.

## Known limitations

- Role outputs are stubbed provider payloads, not live model calls.
- Council thresholds are inherited from 5.2 and still need calibration on real batches.
- Critique is advisory; Phase 7 is expected to add founder review and feedback-loop integration.
