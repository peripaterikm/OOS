# 6.1 Deterministic Anti-Pattern Checks

## Goal

Add cheap deterministic checks for common weak idea patterns before expensive AI council critique.

## Context

Roadmap 5.2 compares ideation modes and produces preliminary candidate recommendations. Before council critique spends provider-style calls, Phase 6.1 adds rule-based checks for generic or operationally weak idea shapes.

## Inputs

- Pattern-guided ideation records
- Legacy `IdeaVariant` records where compatible
- Roadmap anti-pattern list
- Existing ideation mode comparison genericness penalty concept

## Outputs

- `src/oos/anti_pattern_checks.py`
- `tests/test_anti_pattern_checks.py`
- `tests/test_anti_pattern_checks_acceptance.py`
- README section for deterministic anti-pattern checks

## Key Decisions

- Keep the layer deterministic and standalone.
- Do not wire the layer into `run-signal-batch` yet.
- Preserve source ideas without mutation or deletion.
- Reuse the genericness penalty helper from this module inside ideation mode comparison.

## Alternatives Rejected

- No live LLM/API anti-pattern detector.
- No full council-style critique in 6.1.
- No broad anti-pattern scoring refactor.
- No automatic deletion, rejection, or founder-decision replacement.

## Validation

- Focused anti-pattern checks
- Acceptance tests for no live calls, no orchestrator wiring, deterministic rule shape, and roadmap state
- Full unittest discovery
- `scripts/oos-validate.ps1`
- `scripts/verify.ps1`

## Known Limitations

- Rules are keyword/structure based and intentionally conservative.
- False positives are possible for ideas that use dashboard/chatbot language in a specific valid way.
- Council critique in 6.2 still handles semantic/contextual critique.
