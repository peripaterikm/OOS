# 3.1 — Correction Rollback / Undo Policy Review

**Roadmap:** v2.9 — Output Modes, Source URL Strictness & Correction Recovery
**Item:** 3.1
**Status:** complete
**Phase:** A (policy review only; no implementation)
**Dependencies:** None (v2.8 correction infrastructure is foundation)

## Goal

Evaluate whether undo/rollback of founder decision corrections belongs in OOS. Define minimum safe semantics if in scope. Policy-first; implement only if trivially small.

## Context

Roadmap v2.8 delivered safe replace/amend correction modes, parking lot orphan cleanup, derived artifact rebuild, import history audit trail, CLI correction visibility, and E2E correction workflow validation. The v2.8 correction artifact contract (Section 13.8) lists undo/rollback as a "v2.9+ candidate." v2.8 Appendix F item 1 notes: "Requires correction stack/history navigation; v2.8 builds the foundation."

The audit trail (`import_history.json`) and decision archive (`replaced_decisions/`) now exist, providing the raw material for undo. This review determines whether the time is right to implement undo or whether it should be deferred.

## Key Decisions

### 1. Defer rollback/undo to v2.10+

Rollback/undo is classified as **non-trivial** (>200 lines, new module, multi-artifact surgery). Even the narrowest form (undo-last correction) requires a new `correction_undo.py` module, CLI flag, and 4 files of changes. This exceeds both the "trivial" threshold (≤50 lines, 1 file) and the "small" threshold (50–150 lines, ≤3 files).

### 2. Re-replace workaround is adequate for v2.9

A founder who makes a wrong correction can use the existing `--replace-review-items` flag to re-issue a replacement with the original decisions. This is founder-controlled, uses the existing safe code path, and appends a clear audit entry. No new code is needed.

### 3. Only undo-last (U1) is considered safe

General rollback (undoing any correction in the history) requires correction-stack navigation and is too complex. Run-level undo requires pre-correction snapshots that don't exist. Manual restore from archive is trivially small but error-prone.

### 4. 12 safety requirements defined (U-R1 through U-R12)

If undo is implemented in v2.10+, it must satisfy: explicit CLI flag, fail-closed, all-or-nothing, append-only audit, no silent deletion, no portfolio mutation, no autonomous inference, source URL traceability preserved, deterministic, undo-last only, derived artifacts rebuilt, archive integrity verified.

### 5. Future implementation plan recorded

Section 9 of the policy document provides a concrete v2.10 implementation scope: new `correction_undo.py` module, CLI flag, 20+ tests, and design pre-work items (pre-correction snapshots, undo `CorrectionEntry` field, CLI output, manifest integration, replace-all interaction).

## Alternatives Rejected

| Alternative | Reason |
|-------------|--------|
| Implement undo-last now (v2.9) | Exceeds implementation thresholds (~200 lines, 4 files, new module). v2.9 is deferred-item-closure, not feature expansion. |
| Implement general rollback (U2) | Requires correction-stack navigation. Unsafe without formal correction graph validation. |
| Implement run-level undo (U3) | Requires pre-correction snapshots that don't exist. |
| Implement manual restore (U4) | Trivially small but error-prone. Not a real undo solution. |
| No policy review — skip item 3.1 | The roadmap explicitly requires this review. The policy document serves as the v2.10 specification. |
| Implement silent undo without audit trail | Violates append-only audit contract. Every state mutation must be recorded. |

## Known Limitations

- The re-replace workaround requires the founder to manually identify and provide the original decisions file. This is less convenient than a one-command undo but is safe and testable.
- If the `replaced_decisions/` archive is deleted or corrupted, undo would fail. This is acceptable — the archive is the source of truth for pre-correction state.
- Multi-step undo is not covered by this policy. It requires correction-stack navigation that is a v2.11+ concern.
