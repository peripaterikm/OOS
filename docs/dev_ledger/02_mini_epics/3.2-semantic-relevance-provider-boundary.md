# 3.2 Semantic Relevance Provider Boundary

## Goal

Add a clean provider boundary for future semantic relevance scoring without enabling embeddings, vector storage, external APIs, or live model calls.

## Scope

- Added `src/oos/semantic_relevance.py` with semantic relevance input/result dataclasses, provider protocol, disabled provider, keyword stub provider, and provider factory.
- Added optional semantic relevance diagnostic fields to Scoring model v2 input/breakdown.
- Preserved embeddings-disabled scoring as the default behavior.
- Added focused tests for disabled/default behavior, keyword-stub behavior, factory behavior, and scoring diagnostics.

## Out Of Scope

- No real embeddings.
- No vector database.
- No local model inference.
- No OpenAI, Claude, or external embedding API calls.
- No LLM provider contracts or prompt safety work from item 4.1.
- No changes to live collection defaults.

## Provider Behavior

- `disabled` is the default semantic relevance provider and reports `score = None`, `is_available = false`, `embeddings_used = false`, and `external_calls_made = false`.
- `keyword_stub` is deterministic local preview/test support only and uses topic keywords for `ai_cfo_smb`.
- Unknown provider IDs fail deterministically with `ValueError`.

## Scoring Behavior

- Scoring model v2 remains `signal_scoring_v2_embeddings_disabled` by default.
- Disabled or absent semantic relevance records `semantic_relevance_score = 0.0` and leaves final scoring unchanged.
- Explicit available semantic relevance scores are recorded as diagnostics and explained as `diagnostic_only`; they do not alter the current formula.
