## Goal

Replace fragile Unicode symbols in CLI output where they cause CP1251/Windows terminal breakage, and ensure readable output on common Windows terminals without losing information.

## Context

The `weekly-cycle-status-v2` command and other CLI outputs used Unicode characters (e.g., ✅, ❌, ⚠) that rendered as mojibake or blank squares on Windows terminals configured for CP1251 or other non-UTF-8 code pages. The v2.7 final checkpoint recorded this as a known deferred item.

## What Was Deferred

### `--utf8` Flag → v2.9+

The `--utf8` opt-in flag for forcing Unicode output on UTF-8-capable terminals was **not implemented** in v2.8. Deferred to v2.9+.

Rationale: The v2.8 goal was to ensure safe defaults. The `--utf8` flag is a future enhancement for terminals known to support UTF-8. In v2.8, all output is ASCII-safe by default, which is the safer baseline.

## Key Decisions

1. **ASCII-safe by default, no flag needed.** The safest approach for Windows CP1251/CP1252 terminals is to always use ASCII-safe markers. A `--utf8` flag would require the user to opt in and know their terminal supports it — this adds complexity for marginal benefit until v2.9+.
2. **Replace, don't conditionalize.** Rather than adding conditional output logic, all Unicode symbols were replaced with ASCII equivalents. This simplifies the code and eliminates the risk of accidental mojibake.
3. **No business logic changes.** The ASCII-safe replacements are purely cosmetic. Decision values, counts, IDs, and all semantic content are unchanged.

## Alternatives Rejected

- **Per-terminal detection**: Unreliable on Windows; CP1251 detection is fragile.
- **Always UTF-8**: Breaks on CP1251-configured terminals, which is the exact problem this item fixes.
- **Double output (ASCII + Unicode)**: Bloat; the ASCII equivalents are equally clear.
