# Mini Epic 3.1 — Founder Decision Re-Import Policy Review / Safe Replace Mode (Phase A)

**Roadmap:** v2.7 item 3.1
**Date:** 2026-05-08
**Status:** complete (Phase A — policy review only; Phase B deferred to v2.8+)

## Goal

Review the current founder decision import idempotency behavior, document the intended semantics, and decide whether to implement a safe replace/amend mode in v2.7 or defer it to v2.8+.

## Decision

**DEFER to v2.8+.** No replace/amend mode is implemented in v2.7. Roadmap item 3.1 is complete as policy review only.

## Scope (Phase A — Executed)

- Read and analyzed current import behavior in [`src/oos/founder_decision_import.py`](../../src/oos/founder_decision_import.py).
- Analyzed `validate_founder_decision_inputs()` — duplicate `review_item_id` rejection at lines 300-311.
- Analyzed `_merge_decisions()` — replaces by `opportunity_id` at lines 720-730 (but unreachable due to validation).
- Analyzed `_merge_parking_lot_records()` — deduplicates by `record_id` but never removes orphaned records (lines 733-743). This is the critical gap that makes safe replace non-trivial.
- Analyzed downstream artifact rebuild logic: feedback mappings and preference profile rebuild from scratch (safe); parking lot records merge with existing (never remove old records — unsafe for replace).
- Enumerated 13 safety requirements (R1–R13) that any replace/amend mode must satisfy.
- Evaluated 5 options (A–E).
- Recommended Option E: Defer to v2.8+.
- Created authoritative policy document at [`docs/decisions/founder_decision_reimport_policy.md`](../../docs/decisions/founder_decision_reimport_policy.md).

## Scope (Phase B — Deferred)

Not implemented in v2.7. See v2.8 hook note in the policy document for full implementation plan.

## Design Decisions

1. **Defer, don't implement.** The v2.7 roadmap explicitly states "hardening, not feature expansion." A replace mode is feature expansion. The "hardening" gap (placeholder URNs) was already addressed in items 1.1–2.1. Re-import semantics are a workflow convenience, not a hardening defect.

2. **Parking lot record cleanup is the critical blocker.** The existing `_merge_parking_lot_records()` function adds/updates records but never removes them. A PARK→PROMOTE replacement would leave an orphaned `ParkingLotRecord` with a `source_decision_id` pointing to a deleted decision. Fixing this requires a new `_remove_orphaned_parking_lot_records()` function, which touches the artifact writer and `parking_lot.py`.

3. **The existing workaround is sufficient for v2.7.** A founder who needs to correct a decision can re-run `run-weekly-cycle-v2` with the same input signals and the corrected decisions file. This is a full re-run but is safe, deterministic, and preserves traceability.

4. **The policy review itself has lasting value.** The 13 safety requirements (R1–R13) and the v2.8 implementation plan are recorded in the policy document and will inform the actual implementation.
