# 1.2 Parking Lot Orphan Cleanup and Derived Artifact Rebuild Model

## Goal

Implement deterministic helper logic for parking lot orphan cleanup and derived artifact rebuild planning. This item provides the internal primitives that item 1.3 (Safe replace/amend) can use when correction behavior is enabled.

## Context

The current parking lot module (`src/oos/parking_lot.py`) builds records from PARK/REVISIT_LATER decisions but has no mechanism to remove or supersede records. The `_merge_parking_lot_records()` helper in `founder_decision_import.py` only appends new records — it never removes old ones. If a founder replaces a PARK decision with a PROMOTE decision (in a future replace mode), the old `ParkingLotRecord` becomes an orphan, referencing a decision that no longer exists.

This item:
- Creates deterministic identification of orphaned parking lot records.
- Produces cleanup reports without deleting files.
- Rebuilds derived artifacts (feedback mappings, preference profile, parking lot records) from active founder decisions.
- Validates rebuild inputs for source URL traceability.
- Does NOT expose CLI flags, enable replace/amend, or change default import behavior.

## Key Decisions

1. **New module: `src/oos/decision_correction_rebuild.py`.** Preferred over modifying `parking_lot.py` to keep concerns separated. The rebuild module composes existing functions rather than duplicating logic.
2. **Source-decision reference policy: `ParkingLotRecord.source_decision_id` references `FounderDecisionV2.decision_id`.** An orphan is any record whose `source_decision_id` does not appear in the active decisions set. This is the deterministic policy — no guessing.
3. **No file writes.** All functions return data structures and reports. The caller (item 1.3) is responsible for persisting.
4. **Deterministic rebuild order follows contract Section 8.1:** feedback mappings → preference profile → parking lot cleanup + rebuild.
5. **Fail-closed on source URL issues.** Missing source URLs and `urn:oos:*` placeholders are hard errors that block rebuild.
6. **No CLI flags.** No replace/amend behavior. No founder_decision_import.py changes. Default reject-on-reimport behavior preserved.

## Models

| Model | Purpose |
|---|---|
| `ParkingLotCleanupResult` | Result of orphan identification (counts, IDs, no file deletion) |
| `DerivedArtifactRebuildPlan` | Pre-execution plan describing expected rebuild outcomes |
| `DerivedArtifactRebuildResult` | Post-execution result with rebuilt data + cleanup report |

## Functions

| Function | Purpose |
|---|---|
| `identify_orphaned_parking_lot_records()` | Identify orphaned vs active records |
| `cleanup_orphaned_parking_lot_records()` | Clean orphans, return retained records + report |
| `validate_rebuild_inputs()` | Validate decisions + parking lot records pre-rebuild |
| `plan_derived_artifact_rebuild()` | Produce rebuild plan without execution |
| `rebuild_founder_decision_derived_artifacts()` | Execute full rebuild, return rebuilt data |

## Alternatives Rejected

- **Modifying `parking_lot.py` in-place.** Rejected — the cleanup logic is a separate concern (correction workflow) from the parking lot's core function (building and matching records).
- **Duplicating `build_parking_lot_records()` logic.** Rejected — the rebuild module imports and composes existing functions.
- **Adding `--replace`/`--amend` behavior in this item.** Rejected — explicitly out of scope per roadmap; deferred to item 1.3.
