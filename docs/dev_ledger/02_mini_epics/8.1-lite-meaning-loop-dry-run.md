## Adapter Behavior

`CandidateSignal` records are converted into deterministic meaning-loop-compatible records with canonical signal JSONL preview fields:

- `signal_id`
- `captured_at`
- `source_type`
- `title`
- `text`
- `source_ref`

Source Intelligence-specific fields are preserved in metadata and the dry-run traceability map rather than forced into the existing meaning-loop schema.

## CLI Behavior

The weekly discovery CLI now supports:

```powershell
python -m oos.cli run-discovery-weekly --topic ai_cfo_smb --project-root . --include-meaning-loop-dry-run
```

The flag is opt-in. Default weekly discovery behavior remains offline and unchanged when the flag is absent.

## Output Artifacts

When enabled, each discovery run writes:

```text
artifacts/discovery_runs/<run_id>/meaning_loop_dry_run.json
artifacts/discovery_runs/<run_id>/meaning_loop_dry_run.md
```

The JSON report includes counts, adapted records, canonical signal JSONL previews, compatibility status, unsupported fields, limitations, next integration targets, and traceability.

The Markdown report includes Summary, Candidate signals adapted, Downstream artifacts / compatibility status, Traceability, Limitations, and Recommended next steps.

## Traceability Behavior

The dry-run report preserves the chain:

```text
CandidateSignal -> RawEvidence evidence_id -> source_url
```

The traceability map also preserves `source_id`, `source_type`, `topic_id`, and `query_kind`.

## Compatibility Status

Compatibility status is `adapter_only`.

The implementation intentionally does not invoke the production v2.2 meaning loop. It produces deterministic adapted records and reports the remaining integration seams instead of overclaiming full downstream integration.
