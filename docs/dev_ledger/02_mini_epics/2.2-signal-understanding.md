# 2.2 Signal Understanding

## Goal

Define structured signal meaning extraction and quality scoring with provider/stub boundary.

## Context

OOS needs signal-level meaning before semantic clustering and opportunity framing.

## Inputs

- Canonical signals
- AI metadata contract
- Stubbed provider payloads

## Outputs

- `src/oos/signal_understanding.py`
- Meaning and quality score records
- 80% validity fallback rule
- Focused tests

## Key Decisions

- No live LLM calls.
- Invalid per-signal provider outputs fall back to `analysis_unavailable`.
- Duplicate canonical set can avoid denominator inflation.

## Alternatives Rejected

- Replacing existing weak/noise routing.
- Crashing the whole batch for one invalid provider item.

## Validation

- Signal understanding tests.
- Dedup, AI contracts, real batch, and weekly cycle tests.
- Full unittest discovery and `scripts/verify.ps1`.

## Known Limitations

- The provider is stubbed; no real model quality is measured yet.

## Next Step

Use structured signal meaning as optional context for clustering and later stages.
