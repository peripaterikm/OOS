# 5.1 Pattern-Guided Ideation

## Goal

Make LLM-style ideation the primary meaningful idea generation layer while using product patterns to avoid repetitive generic SaaS output.

## Context

Roadmap v2.2 had opportunity cards and a deterministic opportunity quality gate. The next layer needed structured idea variants grounded in opportunities, with product-shape diversity and traceability.

## Inputs

- `OpportunityCard` records from opportunity framing
- Opportunity gate outputs as upstream readiness signal
- Product pattern library
- Stubbed provider payloads
- Existing AI metadata contract

## Outputs

- `src/oos/pattern_guided_ideation.py`
- Product pattern library with seven required product shapes
- `ProductPattern`, `PatternGuidedIdea`, and `PatternGuidedIdeationResult`
- Provider/stub boundary for future LLM ideation
- Low-diversity warning and heuristic fallback behavior
- Focused and acceptance tests

## Key Decisions

- Keep this layer standalone and do not wire it into `run-signal-batch` yet.
- Treat provider output as `llm_assisted` only after validation.
- Label fallback ideas as `heuristic_fallback_after_llm_failure`.
- Preserve linked opportunity IDs and signal IDs on every idea.

## Alternatives Rejected

- Adding live LLM/API calls before comparison and production integration.
- Implementing ideation mode comparison in this mini-epic.
- Replacing existing heuristic ideation or downstream v1 pipeline behavior.

## Validation

- Pattern-guided ideation focused tests.
- Acceptance guards for no live calls, no orchestrator wiring, fallback labeling, and product-pattern diversity.
- Existing ideation flag/evaluation tests.
- Full unittest discovery and `scripts/verify.ps1`.

## Known Limitations

- Provider output is stubbed; real LLM quality is not measured yet.
- Mode comparison, ranking, and weighted evaluation are deferred to item `5.2`.

## Next Step

Implement item `5.2`: ideation mode comparison with weighted metrics.
