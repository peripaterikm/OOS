# 3.2 — Replace-All Mode Policy Review

**Roadmap:** v2.9 — Output Modes, Source URL Strictness & Correction Recovery
**Item:** 3.2
**Status:** complete
**Phase:** A (policy review only; no implementation)
**Dependencies:** None (v2.8 correction infrastructure is foundation)

## Goal

Evaluate whether `--replace-all` mode is safe and needed. Prefer deferral unless a clear real-run need exists. Policy-first; implement only if trivially small.

## Context

Roadmap v2.8 delivered safe replace/amend correction modes (`--replace-review-items`, `--amend-notes-only`), parking lot orphan cleanup, derived artifact rebuild, import history audit trail, and E2E correction workflow validation. The v2.8 correction artifact contract Section 4.4 documents `--replace-all` as "NOT recommended" and "not required for v2.8," noting that it is coarse and violates R8 (silent deletion of unrelated decisions).

The v2.7 re-import policy (Option C) also evaluated `--replace-all` and concluded: "Too coarse. Not recommended."

This review determines whether the time is right to implement `--replace-all` or whether it should remain deferred.

## Key Decisions

### 1. Defer `--replace-all` to v2.10+

`--replace-all` is classified as **not needed for v2.9**. Even though the implementation estimate (~80 lines, 2 files, 15+ tests) qualifies as "small" (50–150 lines, ≤3 files), there is zero demonstrated real-run need. The existing `--replace-review-items` mode and the "re-run the weekly cycle" workaround cover all observed correction use cases.

### 2. Replace-all violates R8 by design

The v2.8 correction artifact contract Section 4.4 explicitly states `--replace-all` "violates R8 (silent deletion of unrelated decisions) unless the founder carefully crafts the new decisions file to include all previously decided opportunities." The surgical `--replace-review-items` mode satisfies R8 by targeting only listed items. Replace-all would need strict-mode completeness validation (RA-R6) to mitigate this risk.

### 3. Re-run workaround is safer and already exists

A founder who needs wholesale decision replacement can re-run `run-weekly-cycle-v2` with the corrected decisions file. This creates a new run with clean artifacts, preserves the old run intact, and requires zero new code. The marginal convenience of in-place replace-all does not justify the safety machinery needed.

### 4. 13 safety requirements defined (RA-R1 through RA-R13)

If `--replace-all` is implemented in v2.10+, it must satisfy: explicit CLI flag, fail-closed, all-or-nothing, dry-run/plan mode first, confirm-step prompt, strict completeness check (reject incomplete replacement files), archive all replaced decisions, append-only audit, no portfolio mutation, no autonomous inference, source URL traceability preserved, deterministic, derived artifacts rebuilt.

Five of these (RA-R4 through RA-R6: dry-run, confirm-step, strict mode) are additional safety requirements beyond what `--replace-review-items` needs — they exist specifically to mitigate the silent-deletion risk.

### 5. Future implementation plan recorded

Section 9 of the policy document provides a concrete v2.10 implementation scope: modify `founder_decision_import.py` (~50 lines for `_import_replace_all_mode()`), modify `cli.py` (~15 lines for `--replace-all` and `--dry-run` flags), and add 15+ tests. Design pre-work items identified: strict vs. lenient completeness check, interaction with undo-last, dry-run output format, parking lot rebuild reporting, and `correction_mode` field value.

## Alternatives Rejected

| Alternative | Reason |
|-------------|--------|
| Implement replace-all now (v2.9) | No demonstrated real-run need. Requires ~80 lines of safety machinery (dry-run, confirm-step, strict mode). v2.9 is deferred-item-closure, not feature expansion. |
| Implement replace-all without strict mode | Silent deletion of omitted decisions is a critical risk. Without strict mode (RA-R6), replace-all is unsafe. |
| Implement replace-all without dry-run | Founder cannot preview what would be replaced. Violates the principle of explicit founder control. |
| Implement replace-all without confirm-step | Accidental execution risk. Every correction mode requires explicit founder opt-in. |
| Remove replace-all from the contract entirely | The contract is the authoritative specification. Removing it would lose the design reference. Keep as deferred for future reference. |
| No policy review — skip item 3.2 | The roadmap explicitly requires this review. The policy document serves as the v2.10 specification and closes the deferred item from v2.8. |

## Known Limitations

- The re-run workaround creates a new run directory rather than editing in-place. This is slightly less convenient but safer and preserves the old run's audit trail.
- If `--replace-all` is implemented in v2.10+, it should require the undo-last feature (3.1) to be implemented first, since undo-last must handle replace-all entries.
- Strict completeness mode (RA-R6) means replace-all cannot be used to delete decisions. Deletion must be done via explicit `--replace-review-items` with a KILL decision. This is intentional — silent deletion is the primary risk.
