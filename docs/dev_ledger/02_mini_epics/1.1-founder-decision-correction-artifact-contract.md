# 1.1 Founder Decision Correction Artifact Contract

## Goal

Define the canonical founder decision correction contract: replace/amend semantics, artifact rewrite rules, no-partial-write behavior, import history/audit fields, and the source URL traceability guarantee that must survive any correction.

## Context

The v2.7 system rejects re-import of decisions for already-decided opportunities (fail-closed, idempotent). The v2.7 re-import policy document (`docs/decisions/founder_decision_reimport_policy.md`) recorded 13 safety requirements (R1–R13) and recommended Option B (`--replace-review-items`) for surgical replacement. This contract item formalizes those requirements into an implementable specification before any code is written.

This is a **docs-only** item. No source code, tests, or artifacts are modified.

## Key Decisions

1. **Contract-first.** No implementation. The contract defines what items 1.2 and 1.3 must satisfy.
2. **Four correction modes defined.** Mode 0 (reject-on-reimport, current), Mode 1 (replace-review-items, surgical), Mode 2 (amend-notes-only, notes/reasons only), Mode 3 (replace-all, documented but not recommended).
3. **`--replace-all` is documented but not required for v2.8.** It is coarse and violates R8 unless carefully justified. Documented for completeness; may be revisited in v2.9+.
4. **Parking lot orphan cleanup is removal, not supersession markers.** Orphaned records are removed from active `parking_lot_records.json`. Supersession is tracked in `import_history.json`.
5. **Checksums are recommended but not mandatory.** If infeasible, omit with a `warnings` note.
6. **`amend-notes-only` allows optional `reason_categories` update.** If reason categories change but decision value stays the same, this is still an amend.
7. **`write-then-rename` atomicity is recommended but not required for v2.8.** The existing direct-write pattern is acceptable if pre-validation and post-validation pass.
8. **Source URL traceability must pass after every correction.** Zero `urn:oos:*` placeholders. Hard requirement.

## Alternatives Rejected

- **Implementing replace/amend inline.** Rejected — this item is contract-first per the roadmap specification.
- **Adding `--replace-all` as a required mode.** Rejected — too coarse, violates R8. Documented but deferred.
- **Using supersession markers in parking lot records.** Rejected — adds complexity; import history already records supersession.
- **Mandatory checksums.** Rejected — optional for v2.8 to keep scope manageable.
- **Separate `CorrectionEntry` per replaced decision.** Rejected — one entry per correction batch simplifies the audit trail.

## Known Limitations

1. `write-then-rename` atomicity is recommended but not designed in detail — left to item 1.3 implementation.
2. The `empty_source_urls_reason` field for exempt inbox items is noted but not fully specified — left to item 1.3.
3. `--replace-all` is documented but its implementation scope is not estimated.
4. Checksum computation overhead for large artifacts is not benchmarked.
5. The contract assumes `manifest.json` can carry `replaced_decision_ids` and `amended_decision_ids` — this field addition is part of item 1.3 scope.
