# 3.2 Contradiction Detection

## Goal

Surface contradictions and merge candidates before opportunity framing while preserving all source signals.

## Context

Signals in the same context can disagree about pain, workaround, urgency, user need, or trust in the current solution.

## Inputs

- Canonical signals
- Optional signal understanding records
- Optional semantic clusters
- Stubbed provider payloads

## Outputs

- `src/oos/contradiction_detection.py`
- `ContradictionReport`, `ContradictionRecord`, and `MergeCandidate`
- Focused and acceptance tests

## Key Decisions

- Merge candidates are suggestions only; no auto-merge.
- Invalid provider records are rejected while valid records can survive.
- Keep the layer standalone for now.

## Alternatives Rejected

- Auto-deleting or auto-merging source signals.
- Wiring contradiction detection into real batch flow prematurely.

## Validation

- Contradiction detection focused tests.
- Acceptance guards for no live calls, no auto-merge/delete, and no orchestrator wiring.
- Full unittest discovery and `scripts/verify.ps1`.

## Known Limitations

- Contradictions are provider/stub driven; no live semantic model yet.

## Next Step

Use clusters and contradiction context for opportunity framing.
